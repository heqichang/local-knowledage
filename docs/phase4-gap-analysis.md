## 详细分析

### 1. 对话历史持久化 ✅

**已实现：**
- `Conversation` / `Message` 模型定义完整
- 消息（user + assistant）在对话过程中持久化到 SQLite
- 切换对话时通过 `GET /conversations/{id}/messages` 恢复历史
- 引用来源以 JSON 格式存储在 `message.references` 字段

**无遗漏。**

---

### 2. 对话列表管理 ⚠️

**已实现：**
- 对话列表查询（按 `updated_at` 倒序）
- 新建对话
- 删除对话（级联删除消息）
- 对话标题自动生成（首条消息前 50 字符）
- 前端侧边栏展示对话列表、切换、删除

**未实现：**

| 缺失功能 | 需求来源 | 影响 |
|----------|----------|------|
| **重命名对话** | 需求 3.4："新建/删除/重命名对话" | 用户无法手动修改对话标题 |
| **更新对话（PUT）** | API 规范："PUT /{id} 更新（标题、关联知识库）" | 无法修改关联知识库 |
| **对话关联知识库选择器** | 需求 3.4："选择关联知识库" | 聊天时无法指定/切换知识库 |

**需要补充的工作：**

1. 后端：添加 `PUT /api/v1/conversations/{id}` 接口
   ```python
   class ConversationUpdate(BaseModel):
       title: str | None = None
       knowledge_base_ids: list[int] | None = None
   ```

2. 前端：对话列表项增加重命名交互（双击编辑 or 右键菜单）

3. 前端：对话界面顶部增加知识库选择器，发送消息时携带 `kb_ids`

---

### 3. 多轮对话上下文 ✅

**已实现：**
- `_get_chat_history()` 获取最近 N 轮对话
- `settings.CHAT_HISTORY_ROUNDS = 5` 可通过环境变量配置
- 历史消息传入 RAG 服务的 `chat_history` 参数

**无遗漏。**

**需要补充的工作：**

1. 前端：上传区域支持文件夹选择（`webkitdirectory` 属性）
2. 前端：批量上传进度条（每个文件独立状态）
3. 前端：批量上传结果汇总（成功/失败数量）
4. 后端：确认多文件并发处理的稳定性

---

### 6. 错误处理与边界情况 ⚠️

**已实现：**
- 流式对话中检索失败/生成失败有 SSE error 事件
- 前端 catch 异常显示"发送失败，请重试"
- 后端 404 对话不存在的处理

**未实现：**

| 场景 | 当前行为 | 期望行为 |
|------|----------|----------|
| Ollama 服务未启动 | 流式报错但信息不友好 | 明确提示"Ollama 未连接"并引导设置 |
| 知识库为空（无文档） | 检索返回空，AI 可能瞎答 | 提示"当前知识库无文档，请先上传" |
| 网络断开 | fetch 抛异常 | 全局 toast 提示网络错误 |
| 消息发送中刷新页面 | 流式中断，消息可能不完整 | 恢复时标记未完成消息 |
| 对话删除确认 | 直接删除无确认 | 二次确认弹窗 |
| 超长消息 | 无限制 | 前端输入长度限制 + 后端校验 |

**需要补充的工作：**

1. 前端：全局错误 toast 组件（网络错误、服务不可用）
2. 前端：删除对话二次确认弹窗
3. 前端：空知识库状态提示
4. 后端：Ollama 连接失败时返回明确错误码
5. 前端：消息输入框长度限制

## 优先级建议

按用户体验影响排序：

1. **设置页面**（高优先级）—— 用户无法切换模型或调整参数，影响基本可用性
2. **对话重命名 + 知识库选择器**（高优先级）—— 对话管理核心交互缺失
3. **错误处理完善**（中优先级）—— 影响使用体验的健壮性
4. **批量导入**（低优先级）—— 当前单文件上传可用，批量为锦上添花
