# Markdown 编辑器 - 产品需求文档

## 概述
- **Summary**: 在现有「本地知识库」项目中新增 Markdown 编辑器功能，支持在线创建和编辑 Markdown 笔记，自动生成分段和向量，参与检索与 RAG 问答。
- **Purpose**: 提供更便捷的文档录入方式，无需先在本地编写 MD 文件再上传，直接在 Web 端即可创建和编辑知识库文档。
- **Target Users**: 需要将本地知识整理入库、快速记录笔记、编辑现有 Markdown 文档的用户。

## 目标
- **Goal 1**: 支持在线创建 Markdown 笔记，保存后自动分段和 Embedding 入库
- **Goal 2**: 支持编辑已有的 Markdown 文档（在线创建的或上传的 `file_type = md`）
- **Goal 3**: 提供分屏所见即所得的编辑体验，左侧源码编辑，右侧实时预览
- **Goal 4**: 保存文档后自动重新生成分段和向量，确保检索数据与文档内容同步

## 非目标 (Out of Scope)
- 自动保存草稿功能
- 版本历史记录和回滚
- 多人协作编辑
- 图片粘贴上传和附件管理
- 富文本 WYSIWYG 模式（Notion 式块编辑）
- 评论功能

## 背景与上下文
- **现有系统**: 项目已实现知识库管理、文件上传、文档处理、向量检索和 RAG 问答功能
- **技术栈**:
  - 后端: FastAPI + SQLAlchemy 2.0 异步 + Alembic
  - 前端: React + Vite + TypeScript + TanStack Query + Zustand
  - 向量数据库: Chroma
- **现有文档流程**: 用户上传文件 → 后台解析 → 文本分段 → 生成向量 → 存储到 Chroma
- **需求来源**: 用户希望能直接在 Web 端编辑 Markdown 文档，无需本地编写后再上传

## 功能需求

### FR-1: 在线创建 Markdown 笔记
- 用户可以在指定知识库中创建新的 Markdown 笔记
- 提供默认的笔记标题（如 `未命名笔记.md`），用户可修改
- 笔记内容为空或全空白时不允许保存
- 保存后自动触发文档处理流程（分段 + Embedding）

### FR-2: 编辑已有 Markdown 文档
- 支持查看和编辑 `file_type = md` 的文档
- 编辑器加载时自动获取文档内容
- 编辑保存后自动重新处理文档（清除旧分段和向量，重新生成）
- 编辑页面显示当前文档的处理状态

### FR-3: 分屏所见即所得编辑器
- 左侧: CodeMirror 6 源码编辑器
- 右侧: react-markdown 实时预览
- 支持代码块语法高亮（Shiki 或 highlight.js）
- 支持 KaTeX 数学公式渲染
- 编辑器和预览区域同步滚动（可选增强）

### FR-4: 顶部工具栏
- **保存按钮**: 手动保存当前文档
- **撤销/重做按钮**: 编辑历史操作
- **格式快捷按钮**:
  - 加粗 (Ctrl+B)
  - 斜体 (Ctrl+I)
  - 标题 (H1/H2/H3)
  - 有序/无序列表
  - 链接
  - 代码块
  - 行内公式/块级公式
- **分屏切换**: 仅源码 / 仅预览 / 分屏模式
- **返回按钮**: 返回到知识库文档列表

### FR-5: 后端 API 支持
- 新增创建在线笔记的 API 端点
- 新增获取/更新 Markdown 文档内容的 API 端点
- 文档内容变更后自动重新触发处理流程
- 权限验证（确保用户只能访问自己的知识库文档）

## 非功能需求

### NFR-1: 前端性能
- 编辑器初始化时间 < 500ms
- 实时预览更新延迟 < 100ms
- 支持至少 10,000 行的 Markdown 文档流畅编辑

### NFR-2: 后端性能
- 文档内容读取和保存响应时间 < 1s
- 文档重新处理在后台异步进行，不阻塞前端响应

### NFR-3: 数据一致性
- 保存成功后，数据库中的文档记录必须已更新
- 重新处理时，旧的分段和向量必须完全清除后再生成新的
- 文档的 `file_hash` 在内容变更时必须更新

### NFR-4: 代码规范
- 遵循项目现有的编码规范和架构模式
- 前端使用自定义 Hook + TanStack Query 管理数据
- 后端使用 Service 层模式处理业务逻辑

## 约束
- **技术约束**: 必须使用项目现有的技术栈
- **依赖约束**: 前端需要新增依赖（CodeMirror 6, react-markdown, 语法高亮库, KaTeX）
- **数据模型约束**: 尽量复用现有的 Document 模型，不做大的改动

## 假设
- 用户编辑的 Markdown 文档可以完整保存到服务器文件系统
- 现有的文件解析和分段逻辑可以直接复用
- 现有的向量生成和存储逻辑可以直接复用

## 验收标准

### AC-1: 在线创建笔记
- **Given**: 用户已进入某个知识库的文档列表页面
- **When**: 用户点击「新建笔记」按钮，填写内容后点击保存
- **Then**: 
  - 系统创建一个新的 Markdown 文档记录
  - 文档内容保存到服务器文件系统
  - 触发后台处理流程（分段 + Embedding）
  - 用户可以在文档列表中看到新创建的笔记
- **Verification**: `programmatic`
- **Notes**: 验证 API 返回和数据库记录

### AC-2: 编辑已有笔记
- **Given**: 用户有一个已存在的 Markdown 文档
- **When**: 用户点击文档的「编辑」按钮，修改内容后保存
- **Then**:
  - 编辑器正确加载原文档内容
  - 保存后文件内容更新
  - 文档的 `file_hash` 更新
  - 旧的分段和向量被清除
  - 新的分段和向量被生成
  - 文档状态更新为 `completed`
- **Verification**: `programmatic`

### AC-3: 分屏编辑器体验
- **Given**: 用户打开编辑器
- **When**: 用户在左侧输入 Markdown 内容
- **Then**:
  - 左侧支持 Markdown 语法高亮
  - 右侧实时显示渲染效果
  - 代码块有语法高亮
  - 数学公式（KaTeX）正确渲染
  - 工具栏按钮可正常使用
- **Verification**: `human-judgment`

### AC-4: 工具栏功能
- **Given**: 用户在编辑器中
- **When**: 用户点击工具栏按钮或使用快捷键
- **Then**:
  - 保存按钮正确保存文档
  - 撤销/重做正确工作
  - 格式按钮插入正确的 Markdown 语法
  - 分屏切换功能正常
- **Verification**: `human-judgment`

### AC-5: 文档参与检索和 RAG
- **Given**: 用户已创建或编辑了一个 Markdown 笔记
- **When**: 用户进行搜索或问答，关键词匹配笔记内容
- **Then**: 该笔记的内容能够被检索到并参与 RAG 问答
- **Verification**: `programmatic`
